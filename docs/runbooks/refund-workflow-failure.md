# Runbook: Refund workflow failure

**Симптом:** клієнт каже, що подав запит на повернення, але не бачить
підтвердження, або каже, що йому "вже мали повернути гроші".

**Як виявити:** `refund-status.js?requestId=...` — статус завжди
`owner_review` одразу після Submit (це НЕ баг — жодна причина, включно
з `changed_mind`, не отримує автоматичний legal verdict, §20).

**Важливо для будь-кого, хто читає цей runbook:** якщо клієнт каже "мені
обіцяли повернення" — перевір, що жоден автоматичний текст ЦЕ не казав
(`refund-status.js`'s `note` завжди: "Запит отримано й передано на
перевірку. Це не автоматичне схвалення повернення."). Якщо клієнт
отримав інше формулювання — це ручна комунікація власника, не
automation-баг.

**Як зупинити шкоду:** `download.js` (payment-readiness цикл) уже
звіряє живий `order.status` при кожному запиті — якщо власник вручну
позначить order як `refunded` (через webhook/simulate-pay `action:
"refund"`), доступ до download відкликається миттєво незалежно від
статусу самого refund-request workflow.

**Як відновити:** якщо `refund-submit` повернув 403
`order_email_mismatch` — це навмисна відмова (email заявника не
збігається з order.email), не публікувати деталі про існування order
під іншим email.

**Як replay:** немає — запит або записався (durable), або 4xx/5xx одразу
видно клієнту, і повторний Submit з тим самим orderId+reason у межах
10-хвилинного вікна поверне той самий `requestId`.

**Як перевірити:** `refund-status` повертає узгоджені orderId/reason/status.

**Коли повідомити власника:** КОЖЕН новий refund_request — власник має
самостійно моніторити (жодного push-сповіщення цього циклу, O-11 досі
OPEN); статус `owner_review` — це буквально "чекає на власника".
