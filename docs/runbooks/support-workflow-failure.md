# Runbook: Support workflow failure

**Симптом:** клієнт каже, що надіслав звернення кілька разів і не
впевнений, чи воно збереглось / чи створилось кілька тікетів.

**Як виявити:** `support-status.js?requestId=...` з тим `requestId`,
який показала форма при Submit — якщо клієнт надсилав кілька разів у
межах 10-хвилинного вікна з тим самим текстом, усі спроби повертають
ОДИН і той самий `requestId` (`duplicate:true` на других+ спробах).

**Як зупинити шкоду:** dedup вже запобігає дублюванню тікетів на рівні
API — немає ризику "10 тікетів від одного повторного клацання".

**Як відновити:** якщо клієнт змінив текст звернення (нова реальна
проблема, не той самий клік) — це СВІДОМО новий `requestId` (dedup-хеш
включає текст) — не баг, а правильна поведінка (§19: "не об'єднувати
різні реальні проблеми лише через однаковий email").

**Як replay:** немає окремого retry — `support_request_created` подія
або записалась (успіх), або весь запит впав з 5xx (клієнт бачить
помилку, може спробувати ще раз — dedup-вікно не спрацює на генуїнно
новій спробі після невдачі, бо event не був записаний).

**Як перевірити:** `support-status` повертає `category`, `status`,
masked email, `createdAt` — усе узгоджено з тим, що клієнт надсилав.

**Коли повідомити власника:** SLA відповіді не підтверджено (O-08) —
власник має знати, що жодна автоматика зараз НЕ повідомляє його про
нові тікети (лише Netlify function logs).
